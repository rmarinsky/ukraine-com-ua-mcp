# Бекап DNS-зон і безпечні зміни

Документ описує як працює механізм preconditioned-backup у `ukraine-com-ua-mcp`,
які гарантії він дає, і як його обходити в нестандартних сценаріях.

## Навіщо це потрібно

API `adm.tools` має суттєві обмеження для безпечної автоматизації:

- **Немає транзакцій.** Кожна зміна DNS відбувається окремим викликом
  `dns/record_add`/`record_edit`/`record_delete`. Якщо посередині серії змін
  щось пішло не так, зона лишається у проміжному стані.
- **Немає version semantics / ETag.** API не має способу сказати «онови запис
  тільки якщо він зараз має значення X». Тобто паралельне редагування
  (наприклад, через web UI) може silent-перезаписати ваші зміни.
- **Немає журналу змін.** Не можна попросити «що змінилось за останню годину».

Без додаткових заходів безпеки LLM, який редагує DNS, працює наосліп. Один
неправильний `delete_dns_record` — і відновлювати доводиться по пам'яті.

Це і вирішує preconditioned-backup: **жодна зміна не проходить без свіжого
снапшота поточного стану зони**, і кожен снапшот криптографічно прив'язаний
до того, що зараз стоїть на боці adm.tools.

## Як це працює

```
┌─────────────────────────────────────────────────────────────────┐
│  1. backup_dns_zone(domain_id="123")                            │
│       ↓                                                         │
│     GET dns/records_list(domain_id=123)                         │
│       ↓                                                         │
│     SHA256(canonical(records)) → backup_id                      │
│       ↓                                                         │
│     Запис у файл: ./dns-backups/123-2026-04-29-...-a1b2c3.json  │
│       ↓                                                         │
│     return { backup_id, path, record_count, expires_at }        │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  2. delete_dns_record(domain_id, subdomain_id, backup_id, ...)  │
│       ↓                                                         │
│     Завантажити backup-файл → перевірити saved_at + maxAgeMs    │
│       ↓                                                         │
│     GET dns/records_list(domain_id) → свіжа зона                │
│       ↓                                                         │
│     SHA256(свіжа зона) === backup_id?                           │
│       ↓ так                              ↓ ні                   │
│     POST dns/record_delete                BackupError           │
│       ↓                                                         │
│     return { deleted: true, ... }                               │
└─────────────────────────────────────────────────────────────────┘
```

Канонізація (для стабільного хеша):
- Записи сортуються за полем `id` (числове порівняння, з fallback на лексикографічне).
- Для кожного запису ключі об'єкта сортуються у JSON-серіалізації.
- Поля з `undefined` значеннями ігноруються (відповідає семантиці `JSON.stringify`).

Це гарантує: однаковий вміст зони → однаковий `backup_id`, незалежно від того,
у якому порядку API повертає записи цього разу.

## Робочий процес

### Простий випадок: видалити один запис

```
1. backup_dns_zone(domain_id="123")
   → { backup_id: "a1b2...", record_count: 12, ... }

2. list_dns_records(domain_id="123")
   → знайти потрібний запис, його id = "456"

3. delete_dns_record(
     domain_id="123",
     subdomain_id="456",
     backup_id="a1b2...",
     confirm=true
   )
   → { deleted: true, ... }
```

### Серія змін (наприклад, налаштування пошти)

```
1. backup_dns_zone(domain_id="123") → backup_id_v1

2. create_dns_record(domain_id, type=MX, ..., backup_id=backup_id_v1)
   → MX додано

3. backup_dns_zone(domain_id="123") → backup_id_v2
   (важливо: після кожної зміни беремо новий бекап, бо стара хеш-сума
    більше не співпадатиме)

4. create_dns_record(domain_id, type=TXT, record="@", data="v=spf1 ...", backup_id=backup_id_v2)
   → SPF додано

5. ...і так далі
```

**Альтернатива:** якщо знаєте, що робитимете багато змін підряд — збільшіть TTL:

```bash
ADM_TOOLS_BACKUP_MAX_AGE_MS=3600000  # 1 година
```

Тоді один `backup_dns_zone` на початку покриває всю сесію, але треба пам'ятати,
що між мутаціями зона не повинна змінюватись паралельно.

### Відкат через restore_dns_zone

Якщо щось пішло не так, можна повернути зону до стану з бекапу:

```
restore_dns_zone(
  domain_id="123",
  backup_id="a1b2...",
  confirm=true
)
→ {
  created: 1,    // були в бекапі, видалені після — створено заново
  updated: 2,    // змінилися значення — оновлено
  deleted: 0,    // були додані після бекапу — видалено
  unchanged: 9,  // не змінилися — нічого не робили
  errors: []     // якщо щось впало — звіт по записах
}
```

**Важливо про restore:**

- **Не атомарне.** Якщо replay падає на 5-му записі з 20, попередні 4 зміни
  вже застосовані. Тоді у відповіді буде повний звіт `errors[]`. Зону можна
  було б довідновити вручну або повторним викликом `restore_dns_zone`.
- **Записи отримують нові id.** Якщо запис був видалений після бекапу і
  restore відтворює його — adm.tools видає новий `subdomain_id`. Старий уже
  не існує. Для `update`-операцій id зберігається.
- **Без dry-run.** Поки що не реалізовано: показати diff, не виконуючи.
  Якщо потрібно — створіть issue.

## Конфігурація

Усі змінні середовища опціональні, default-и підібрано для типового
LLM-сценарію.

| Змінна | Default | Що робить |
|--------|---------|-----------|
| `ADM_TOOLS_BACKUP_DIR` | `./dns-backups` | Куди зберігати JSON-файли бекапів. Шлях відносний до робочої директорії MCP-сервера. |
| `ADM_TOOLS_BACKUP_MAX_AGE_MS` | `600000` (10 хв) | Скільки `backup_id` лишається валідним. Після цього — потрібен новий `backup_dns_zone`. |
| `ADM_TOOLS_REQUIRE_BACKUP` | `true` | Якщо `false` — `backup_id` стає опціональним параметром на write tools. Для CI/automation, не для інтерактивної роботи. |

## Структура файлу бекапу

```json
{
  "domain_id": 123,
  "saved_at": 1759084800000,
  "backup_id": "a1b2c3d4e5f6...",
  "records": [
    { "id": 456, "type": "A", "record": "@", "data": "1.2.3.4" },
    { "id": 789, "type": "MX", "record": "@", "data": "mx.ex.com", "priority": 10 },
    ...
  ]
}
```

Назва файлу: `<domain_id>-<ISO timestamp>-<перші 12 символів backup_id>.json`.
ISO timestamp у назві щоб `ls -la` показував зрозумілі дати.

## Troubleshooting

### `Backup not found for domain_id=X backup_id=Y`

Файл не знайдено в директорії бекапів. Можливі причини:
- Видалений вручну (директорія `./dns-backups/`).
- MCP-сервер запущено з іншої робочої директорії — relative path
  `./dns-backups` тепер вказує не туди. Використовуйте абсолютний шлях
  у `ADM_TOOLS_BACKUP_DIR`.
- Бекап ніколи не створювався — забули викликати `backup_dns_zone`.

**Рішення:** викликати `backup_dns_zone` для цього `domain_id` ще раз.

### `Backup expired (older than NNNms)`

Минуло більше, ніж `ADM_TOOLS_BACKUP_MAX_AGE_MS` від часу створення бекапу.
Захист від «бекап двотижневої давнини» — за цей час зона могла змінитись
сотню разів.

**Рішення:** викликати `backup_dns_zone` ще раз — він поверне свіжий `backup_id`.

### `Zone changed since backup_id was issued`

Хеш зараз поточної зони не співпадає з `backup_id`. Це значить, що між
`backup_dns_zone` і вашим write-викликом хтось (інший адмін, інший процес,
веб-інтерфейс) змінив записи.

**Рішення:**
1. Викликати `list_dns_records(domain_id)` і подивитись, що змінилось.
2. Викликати `backup_dns_zone(domain_id)` для нового снапшоту.
3. Повторити мутацію з новим `backup_id`.

### Усе ламається, потрібно швидко зробити автоматизацію

CI-сценарії, де ви точно знаєте, що зону ніхто не правитиме паралельно:

```bash
ADM_TOOLS_REQUIRE_BACKUP=false
```

Тоді `backup_id` стає опціональним. Якщо ви все одно його передасте — він
все одно перевіряється; просто можна тепер вимагати `confirm: true`.

## Обмеження

- **Файлове сховище.** Бекапи лежать на файловій системі MCP-сервера. Якщо
  сервер крашиться між `backup_dns_zone` і мутацією — файл лишається, але
  state не shared між інстансами.
- **Без auto-cleanup.** Старі бекапи не видаляються автоматично, тільки
  TTL використовується для валідації. Періодично чистіть `./dns-backups/`
  або налаштуйте cron.
- **Один бекап на хеш.** Якщо ви робите `backup_dns_zone` коли зона
  не змінилась — отримаєте той самий `backup_id`, але з новим файлом
  на диску (з новим timestamp у назві).
